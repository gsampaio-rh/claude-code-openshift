# ADR-001: Kata Containers for Agent Isolation

**Status:** Accepted (updated: see ADR-017 for bare metal requirement on AWS EC2)
**Date:** 2026-04-08
**Deciders:** Platform Engineering

## Context

AI coding agents execute untrusted code generated by LLMs — installing packages, modifying files, running shell commands. Multiple agents run on shared infrastructure for 5-20 developers. We need isolation that prevents a compromised agent from affecting the host or other agents.

Options considered:

1. **Standard containers (runc)** — Linux namespace isolation only
2. **gVisor** — userspace kernel that intercepts syscalls
3. **Kata Containers** — microVM with dedicated kernel per pod
4. **Full VMs** — traditional virtualization

## Decision

Use **Kata Containers** via the OpenShift Sandboxed Containers Operator.

## Rationale

- **Kata is GA on OpenShift** as a layered product. gVisor is not supported by Red Hat.
- Kata provides **hardware-level isolation** (KVM) — each agent gets its own kernel. Even with `privileged: true` inside the VM, there is no host access (`privileged_without_host_devices=true`).
- Kata supports **Docker-in-sandbox** — agents can spin up databases, dev servers, etc. gVisor has limitations with nested container runtimes.
- The Red Hat AI BYOA blog explicitly references Kata and the upcoming agent-sandbox integration as the isolation strategy.
- From a Kubernetes perspective, Kata pods look like normal pods. Only `runtimeClassName: kata` is added — no changes to application code.

## Trade-offs

- **Requires bare metal or nested virt**: Kata needs KVM. Clusters running on VMs without nested virtualization enabled cannot use Kata. Mitigation: validate in Phase 0, fall back to gVisor if needed.
- **Higher resource overhead than gVisor**: each Kata VM has its own kernel, consuming more memory (~50-100MB overhead per VM vs ~10-20MB for gVisor).
- **Startup latency**: Kata VMs take ~1-2s to start vs ~100ms for runc. Mitigation: acceptable for workspace-lifecycle pods (not request-level).

## Consequences

- OpenShift Sandboxed Containers Operator must be installed
- KataConfig CR must be created to enable Kata runtime on worker nodes
- Coder workspace templates must set `runtimeClassName: kata`
- NetworkPolicy must be configured per-sandbox to restrict egress
